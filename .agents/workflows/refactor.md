# Refactor
**Description:** Рефакторинг существующего кода GLS Admin. Улучшает качество без изменения поведения. Находит типичные проблемы проекта и исправляет их.

## Steps

### Step 1 — Анализ файла
Открыть файл который нужно рефакторить и проверить наличие этих проблем:

**Vue компоненты:**
- [ ] Прямые axios-вызовы в компоненте (должно быть через store/api)
- [ ] Хардкоженные строки вместо `t('key')`
- [ ] Отсутствие состояний `isLoading` / `error`
- [ ] `Options API` вместо `<script setup>`
- [ ] Нетипизированные props

**Laravel:**
- [ ] `Student::count()` или `->count()` для счётчиков в реальном времени
- [ ] Валидация в контроллере вместо Form Request
- [ ] Raw модель в ответе вместо API Resource
- [ ] HTTP-вызов к LMS внутри контроллера (должен быть Job)
- [ ] Отсутствие проверки на дубль webhook (transaction_id)

**API слой (frontend):**
- [ ] URL строки напрямую вместо `endpoints.ts` констант
- [ ] URL без `/v1/` префикса
- [ ] Использование `httpClient.ts` вместо `http.ts`

### Step 2 — Объяснить что найдено
Перечислить все найденные проблемы и объяснить ПОЧЕМУ это проблема.
Не просто "это плохо" — а конкретное последствие (производительность, безопасность, поддержка).

### Step 3 — Показать план изменений
Перед изменением кода показать:
- Что будет изменено
- Что останется без изменений
- Есть ли риск сломать что-то ещё

Получить подтверждение от пользователя.

### Step 4 — Выполнить рефакторинг
Вносить изменения пошагово, не все сразу.
После каждого изменения объяснять что было сделано.

### Step 5 — Обновить связанные файлы
Если рефакторинг затрагивает:
- Переименование → обновить все импорты
- Новые ключи i18n → добавить в все 4 локали
- Новый эндпоинт → добавить в `endpoints.ts`
- Удаление метода → проверить что нигде не используется

### Step 6 — Проверка результата
- [ ] Поведение не изменилось — только структура кода
- [ ] Нет новых TypeScript ошибок
- [ ] Все переводы добавлены
- [ ] Нет неиспользуемых импортов
