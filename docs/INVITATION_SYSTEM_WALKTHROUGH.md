# Рефакторинг процесса приглашения учеников

Этот документ суммирует полную переработку системы приглашений, гарантируя её безопасность, строгую валидацию данных и полное соответствие схеме базы данных проекта Indigo Mental.

## 🎯 Цели проекта
- **Стандартизация**: Замена ручного создания учеников на защищенный процесс регистрации через приглашение.
- **Безопасность**: Исключение дубликатов никнеймов и строгая проверка всех входящих данных.
- **Соответствие БД**: Гарантия того, что все данные из CRM в точности соответствуют таблице `import_students_full`.
- **Централизация**: Использование единого UI-компонента (`InviteLeadModal`) для приглашений как со страницы «Лиды», так и со страницы «Все ученики».

## 🛠 Реализация на Backend (Indigo API)

### 1. Новый API эндпоинт
- **URL**: `POST /v1/recruitment/invite`
- **Контроллер**: [NewStudentsController.php](file:///Users/artsiomhrableuski/Projects/Indigo/app/Http/Controllers/Api/NewStudentsController.php)
- **Модель**: Используется [ImportStudentFull.php](file:///Users/artsiomhrableuski/Projects/Indigo/app/Models/ImportStudentFull.php) для работы с данными.

### 2. Строгие правила бизнес-логики
> [!IMPORTANT]
> **Проверка уникальности никнейма**: Перед вставкой API проверяет поле `nickname` сразу по двум таблицам: `import_students_full` и `recruting_student`. Если никнейм уже занят, возвращается ошибка `409 Conflict`.

- **Маппинг полей (Согласно ТЗ техлида)**:
    - `email` → Email родителя
    - `parent_email` → Email родителя
    - `nickname` → Email ученика (используется как идентификатор)
    - `is_send` = `0` (Начальное состояние)
    - `is_done` = `0` (Начальное состояние)
    - `import_source` = `'crm_teacher_invite'` (Источник)

- **Белый список контрактов**: Разрешены только значения из списка:
    - `contract_399`, `contract_489`, `contract_600`, `contract_689`
